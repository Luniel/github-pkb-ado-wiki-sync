
# This is a page template for drafting a new Behavioral Specification in Azure DevOps Wiki

**IMPORTANT NOTES**
1. Copy the contents of the appropriate template section below into a new Behavioral Specification page.
2. After pasting, remove any escape characters before code blocks if needed.
3. Fill in the necessary details as instructed.
4. Only when the specification is being promoted to `Approved New` should empty and unneeded sections be deleted.
5. This page expresses behavior. It must not be used to define implementation logic, hidden rationale, or capability structure.
6. The `# Dependencies` section is only for true capability dependencies that must be used in implementation. It is not a general related-links section.
7. The `# Implementation Notes` section is optional and non-authoritative. Use it only to preserve specific implementation-level notes or choices requested by Engineering. Do not use it for behavioral clarification, analysis residue, or scope notes.
8. The `# Additional Aspects` section is for genuine supplementary aspect artifacts that apply a distinct governed lens to the same core behavior, such as visual design, accessibility, security, legal, finance/tax, compliance, or other stakeholder-specific concerns. It is not a generic related-links section.
9. Gherkin scenario construction, naming, formatting, table usage, and other scenario-form craftsmanship are governed by the BDD Craftsmanship Standard, not by this template alone.
10. Do not create separate Additional Aspect artifacts merely because a distinct concern can be named. Create them only when doing so materially improves governed concern preservation, stakeholder-specific review, or readability of the core behavioral specification.
