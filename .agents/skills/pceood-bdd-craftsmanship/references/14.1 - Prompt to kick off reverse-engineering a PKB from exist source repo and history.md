
## Prompt to kick off reverse-engineering a PKB from exist source repo and history

The best reusable kickoff prompt is one that prevents drifting too early into target-state thinking, promotion logic, or broad speculative redesign when the real job is to establish a **truthful current-state PKB baseline** for an already implemented system. That sequencing is already supported by the doctrine: first-pass PKB bootstrapping is allowed to use code, history, and other non-authoritative evidence to establish a valid but immature PKB, and current-state specification is a legitimate way to answer what the system actually does before deciding what it should do next.  

### Minimum files to provide

1. **The current source repo ZIP**

   This is non-negotiable. The repo is the main evidence source for actual implemented behavior.

2. **The current authoritative PCEOOD baseline**

   Also non-negotiable, because the whole effort depends on doctrinal ownership, lifecycle separation, and first-pass bootstrapping rules. The doctrine explicitly organizes authority by owned layers and warns against solving ambiguity in the wrong downstream place. 

3. **Historic solution-specific rationale**

   Minimum acceptable form:

   * exported prior chats about the solution, or
   * a structured history file, or
   * design/rationale notes that are actually about that solution

   This matters because the doctrine expects continuity and preserved rationale, not just code archaeology. The current-state / target-state note also assumes behavior often has to be reconstructed from real evidence, not only from polished specs. 

That is the true minimum.

Strongly recommended additional files:

4. **Any existing partial PKB or proto-PKB**

   Even if immature or incomplete, include it. It may already contain useful continuity, naming, or ownership decisions.

5. **One real example PKB**

   Only for structure and craftsmanship conventions. This is helpful because it shows how the doctrine looks in practice, but it must never become source truth for the new solution. 

6. **PKBDD templates**

   * capability template
   * behavioral specification template
   * visual design template

   These help keep page classes disciplined and prevent accidental artifact drift.   

7. **RMF templates**

   Only needed if you expect readiness/completion support surfaces to be created during the same effort.   

My practical recommendation for the minimum set is:

* repo ZIP
* current PCEOOD file
* one solution-history file or exported prior chat thread

If you can provide two more, make them:

* the current PKB or partial PKB, if any
* the exporter-specific or solution-specific prior continuity thread

That gives the best accuracy for the least upload burden.

Here is the prompt I would use for other existing software solutions that do not yet have a PKB:

### Prompt

```markdown
You are beginning PKB establishment work for an existing software solution that currently has **no PKB**.

Your immediate objective is:

**Build out and mature a PKB until it fully and truthfully represents the currently implemented system’s real capabilities and behaviors.**

This is the governing intent for this phase.

We are **not** starting with target-state redesign.
We are **not** starting with speculative future architecture.
We are **not** promoting artifacts prematurely just because they are cleaner.
We are **not** treating implementation artifacts as authoritative requirements.

We are first establishing a **truthful current-state PKB baseline** for the existing implemented system.

## Required doctrinal stance

Use the attached Producore Canonical Engineering Ontology and Operating Doctrine as the governing baseline.

You must preserve the distinctions among:
- ontology
- authority
- structural derivation
- PKBDD craftsmanship
- RMF operating doctrine
- RMF craftsmanship
- engineering realization
- applied notes

You must especially preserve the distinction among:
- implemented behavior
- current-state specification/capture
- target-state intended behavior
- lifecycle promotion / binding authority

Do not collapse them.

## What kind of work this is

This is a **reverse-engineering PKB establishment effort** for an existing implemented solution.

The immediate goal is to create a PKB that:
- truthfully captures the currently implemented system
- preserves real continuity and rationale
- makes implicit behavior explicit
- identifies actual capability ownership
- uses governed approximations where needed
- preserves unresolved meaning honestly
- does not fabricate maturity

Only after that current-state baseline is trustworthy should promotion and future-state evolution become the focus.

## Working method

Follow this sequence:

1. Study the implementation repo directly through filesystem access.
2. Study the governing doctrine in full.
3. Study any available historical decisions/rationale for this specific solution.
4. Study any available example PKB only for structure/craftsmanship conventions, never as source truth for this solution.
5. Infer the actual currently implemented capabilities and behavior surfaces from the evidence.
6. Build or mature the PKB so that those implemented capabilities and behaviors are explicitly and usefully represented.
7. Prefer strengthening current-state capture inside real capability ownership over inventing broad new decomposition.
8. Only add new capability surfaces where the code and evidence clearly force it.
9. Do not silently treat code, tests, readmes, or tickets as authority; use them only as evidence for current-state capture.
10. Preserve rationale, unresolved questions, approximations, and maturity boundaries explicitly.

## Structural rules

1. The PKB is authoritative source material, not casual documentation.
2. The PKB must live inside the source repo.
3. Use Azure DevOps Wiki-style structure:
   - a parent page is a real `.md` file
   - if that page has children, it also has a same-named sibling folder
   - that folder contains the child pages
   - `.order` files preserve ordering where needed
4. Parent pages should use curated child listings rather than `[[_TOSP_]]` by default, unless there is a specific reason not to.
5. Be explicit about artifact class:
   - authoritative capability definition
   - governed approximation
   - current-state behavior capture
   - unresolved/open questions
   - traceability support
   - non-authoritative support surfaces
6. Do not fake mature target-state authority when only current-state capture is justified.
7. Do not erase continuity merely to make the PKB look cleaner.

## Output requirements

At the start of your response, provide a short `Current State` block with:
- Repo Source Status
- Repo Source
- PCEOOD Status
- PCEOOD
- PKB Status
- Current working objective
- What has been verified directly in this step
- What remains uncertain
- Next concrete action
- Working mode

Then do the work.

If concrete PKB changes are warranted, produce a downloadable PKB patch ZIP.
Do not stop at analysis or recommendations.

## Decision rule for this phase

The main question is:

**What currently implemented capability/behavior surfaces are still not fully and cleanly represented in the PKB, and how should the PKB be matured to fix that without fabricating authority or maturity?**
```
